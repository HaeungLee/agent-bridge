# Agent Bridge Commander Handbook

Date: 2026-05-22

Purpose: give a fresh Codex commander enough project state to continue without becoming a subordinate implementation agent.

## Commander Role

Codex is the commander, architect, reviewer, and final gate.

Codex should spend tokens on:

- Foundation design.
- Scope boundaries.
- Task specs for subordinate agents.
- Acceptance criteria and gates.
- Code review and integration decisions.
- Model usefulness evaluation.

Subordinate agents should spend tokens on:

- Narrow implementation tasks.
- Read-only repository reports.
- Benchmark runs.
- First-pass investigations.
- Patch production inside isolated worktrees.

The bridge is not complete until Codex can call subordinate agents through `agent-bridge` with the same discipline as a local tool call.

## Current Operating Model

Until the bridge is fully automated, the human operator may act as the bridge between Codex and external agents.

Expected flow:

1. Codex writes the task spec and prompt.
2. The human or `agent-bridge` runs Gemini, OpenCode/Kimi, DeepSeek, Mimo, Antigravity, or another adapter.
3. The subordinate agent writes artifacts under `.agent/runs/` and appends only to `docs/process/`.
4. Codex reviews compact artifacts before raw logs.
5. Codex decides whether to accept, reject, request another run, or update rules.

Do not let subordinate agents edit the active workspace directly for write-capable tasks. Write-capable work must run in an isolated git worktree or temporary workspace and export a patch.

## Model Routing Memory

`.agent/metrics/model_routing.md` is generated routing memory, not source of truth.

It is currently regenerated by `agent-bridge eval` from run directories containing `verdict.json` and `decision_report.json`. This made early mock or placeholder agents appear as real model evidence. For example, `glm-5.2` entries came from `glm_review` placeholder/mock evaluations, not confirmed GLM model calls.

Policy:

- Treat `model_routing.md` as a generated convenience view.
- Do not rely on it as primary evidence.
- Primary evidence lives in per-run artifacts and process logs.
- Separate real benchmark evidence from mock, smoke, blocked, and placeholder runs.
- Exclude unavailable or uncalled models from routing conclusions.
- Keep generated routing memory only if it helps commander decisions after filtering run types.

Recommended redesign:

- Keep `.agent/metrics/model_routing.md` as generated human-readable notes.
- Add or later generate `.agent/metrics/benchmarks.jsonl` for benchmark evidence.
- Add run classification fields such as `run_kind = mock|smoke|readonly_benchmark|write_patch|review`.
- Add `is_real_model_call = true|false`.
- Add `model_available = true|false|unknown`.
- Generate routing notes only from accepted evidence classes unless explicitly asked for all runs.

## Current Model Observations

Kimi k2.6 through OpenCode/nanoGPT:

- Useful for compact configuration reports.
- Not yet reliable for strict read scope on source or gate-understanding tasks.
- Must be followed by `task gate` or `check-tool-use`.

Gemini and Antigravity:

- Fast and useful for documentation/rules work.
- Historically prone to scope creep and process-log overwrite risk.
- Must receive narrow task specs and append-only instructions.
- Antigravity smoke remains smoke-only because it uses permissive CLI flags and global session-state observation.

DeepSeek v4 Flash/Pro and Mimo v2.5 Pro:

- Candidate models for 1M-context benchmark comparison.
- Need the same read-only benchmark matrix before routing decisions.

## Isolated Worktree Execution Direction

Default write-capable isolation unit: git worktree.

Target flow:

1. Create a temporary git worktree from the active repository base.
2. Run the subordinate agent only in that worktree.
3. Enforce task `read_scope`, `write_scope`, and `forbidden_files`.
4. Capture child CLI output, tool use, changed files, tests, diffstat, and risks.
5. Export `patch.diff` into the run directory.
6. Run gates against the isolated worktree and run artifacts.
7. Leave the active workspace untouched unless Codex explicitly applies the patch.

Dirty active workspace policy:

- Do not block read-only benchmark runs solely because the active workspace is dirty.
- For write-capable execution, choose a base commit or snapshot explicitly.
- Run result checks against the isolated worktree diff, not the active workspace status.
- If user changes are needed in the worktree, require an explicit policy before copying them.

## Review Rubric

Codex evaluates subordinate output using factual criteria:

- Scope discipline: stayed inside allowed files, read scope, write scope, and task phase.
- Artifact quality: produced required summary, decision report, tests, risks, touched files, diffstat, raw logs, and patch when applicable.
- Code quality: minimal, idiomatic, locally consistent, no unrelated abstractions.
- Verification: ran required commands or explained blockers.
- Instruction following: no self-verdict, no self-scoring, no direct commits, no process overwrite.
- Model fitness: suited or unsuited for this class of task.

Do not write subjective self-scores into process logs. Commander verdicts are reserved for Codex/human review.
